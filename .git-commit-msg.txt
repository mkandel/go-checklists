Allow checklist creators to edit items regardless of status

Adds a creator-only override on top of the existing assignee/approver
flows: add, remove, reorder, or directly check/uncheck items at any
status, including complete. Any such edit unconditionally forces the
checklist back to open (emitting a reopened event and notifying the
current assignee), leaving assigned_user_id untouched.

- Migration: soft-delete checklist_items via deleted_at, since
  checklist_events.item_id has a hard FK that would block a real
  DELETE once an item has history.
- Domain: forceReopen helper, AddItem, RemoveItem, ReorderItems,
  SetItemChecked; new ErrNotCreator and ErrInvalidReorder sentinels;
  new event constants item_added, item_removed, items_reordered,
  reopened.
- Store: ChecklistRepo.Save now reconciles c.Items against the DB
  (insert new, update existing, soft-delete removed) instead of
  assuming a fixed pre-existing item set. Get filters out
  soft-deleted items. notificationForEvent handles EventReopened.
- Tests: domain unit tests for creator gating, reopen semantics, and
  position renumbering; Postgres integration tests covering
  persistence, soft-delete survival of the item_id FK, reorder
  persistence, and reopening a complete checklist.
- DESIGN.md updated to document creator overrides and the
  checklist_items soft-delete pattern.
