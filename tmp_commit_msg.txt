Add multi-tenancy: tenant_id scoping across schema, domain, store, API

Add a tenants table and thread tenant_id through users, groups,
templates, checklists, notifications, and checklist_events. Composite
UNIQUE(tenant_id, id) + composite FKs enforce cross-tenant referential
integrity at the database level, closing a latent cross-tenant data
leak in checklist visibility/lookup.

- New TenantRepo with GetSoleTenant, the deliberately-temporary stand-in
  for per-request tenant resolution (errors loudly if tenant count != 1
  rather than silently defaulting).
- Repo methods that take bare IDs (Get, Claim, List, GetByUsername,
  ListForUser, MarkRead, group membership checks, etc.) now take an
  explicit tenantID and filter on it. Methods that take a domain struct
  pointer don't need a separate param since TenantID travels with it.
- main.go auto-provisions a default tenant on startup so existing
  on-prem installs work with zero manual setup.
- handleLogin resolves the sole tenant pre-auth via GetSoleTenant; true
  multi-tenant login (subdomain/host-based resolution) is deferred v2
  scope, documented in DESIGN.md alongside self-service signup,
  per-tenant billing, and Postgres RLS.
- DESIGN.md: new Multi-tenancy section, updated table field lists and
  architecture diagram, v2 SaaS scope called out under Open items.
- Test suites (internal/store/postgres, internal/api) updated to
  bootstrap a test tenant and stamp TenantID on fixtures.

Verified: go build/vet/test (incl. testcontainers-backed Postgres and
API suites), gofmt clean, and a manual smoke test of the full
login -> create -> claim -> check -> complete flow against a real
Postgres instance with the auto-provisioned default tenant.
